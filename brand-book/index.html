Design System

Regras de Componentes

Padrões visuais obrigatórios para garantir a identidade Realizzati em todos os componentes de UI.

01

Efeito Double-Bezel (Chanfrado)

Todos os containers de destaque devem ter borda sutil com opacidade (border: 1px solid rgba(223,209,193,0.2)) e cantos arredondados generosos (border-radius: 2rem). O interior deve conter um elemento ligeiramente menor, simulando o recuo geométrico de portas de armários planejados.

Double-Bezel
02

Botões com Micro-movimento

Botões de ação principais devem ter o texto alinhado à esquerda e o ícone de seta envolto em uma bolha à direita, com transições físicas suaves usando cubic-bezier(0.32, 0.72, 0, 1). O ícone deve se deslocar levemente no hover.

03

Imagens com Gradiente de Profundidade

Toda imagem de ambiente (cozinhas, closets, salas) deve ter um overlay gradiente escuro de baixo para cima, garantindo legibilidade total para textos e logos aplicados por cima. CSS: background: linear-gradient(to top, rgba(29,19,17,0.9) 0%, transparent 60%).

Design System

Auditoria de Consistência

Antes de publicar qualquer comunicação ou interface da Realizzati, responda estas perguntas obrigatórias.

?

Estamos usando jargões genéricos?

Se houver termos como "qualidade garantida" ou "melhor preço", reescreva focando na exatidão técnica e valorização imobiliária.

?

O contraste está legível?

Certifique-se de que nenhum texto em Sand Gold ou off-white esteja aplicado diretamente sobre fundos claros sem overlay escuro de contraste.

?

A mensagem respeita a autonomia do cliente?

Evite gatilhos de urgência agressivos ("Últimas vagas", "Sua chance acaba hoje"). Use CTAs baseadas em progresso: "Iniciar estudo do meu espaço".

Diretrizes de Agente IA

Regras comportamentais obrigatórias para todos os agentes de IA que operam neste workspace.

CRÍTICO · Git & Segurança

Proibição de Push Remoto

É estritamente proibido realizar git push para repositórios remotos sem autorização explícita do desenvolvedor humano. Todo código gerado deve resultar em testes locais e no máximo um git commit semântico local. A revisão e o envio para a nuvem são responsabilidade do desenvolvedor humano.

Frontend

Padrões de Engenharia

  • Mobile-First: Todo componente deve ser desenhado para telas pequenas primeiro, com prefixos responsivos para telas maiores.
  • Tipagem: Código rigorosamente tipado com TypeScript. Tags HTML5 semânticas e atributos aria-*.
  • Modularidade: Componentes complexos devem ser quebrados em partes menores antes de integrar à página principal.
